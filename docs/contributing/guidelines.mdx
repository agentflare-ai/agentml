# Contributing Guidelines

We welcome contributions to AgentML! Help us build the universal language for AI agents.

## Code of Conduct

Be respectful, inclusive, and constructive. We're building a welcoming community around open agent standards.

## Ways to Contribute

1. **Report bugs**: Help us improve stability
2. **Request features**: Propose new capabilities
3. **Write documentation**: Improve guides and examples
4. **Create extensions**: Build custom namespaces
5. **Fix issues**: Submit bug fixes and improvements
6. **Share examples**: Contribute agent templates
7. **Test conformance**: Validate SCXML compliance

## Getting Started

### Fork and Clone

```bash
# Fork the repository on GitHub
# Clone your fork
git clone https://github.com/yourusername/agentml.git
cd agentml

# Or for runtime development
git clone https://github.com/yourusername/agentmlx.git
cd agentmlx
```

### Create a Branch

```bash
git checkout -b feature/your-feature-name
```

### Make Changes

Follow the development guide for setup instructions.

### Commit Changes

Use conventional commit messages:

```bash
git commit -m "feat: Add streaming support to Gemini extension"
git commit -m "fix: Resolve state transition bug in parallel states"
git commit -m "docs: Update quick-start guide with new examples"
git commit -m "test: Add SCXML conformance tests for datamodel"
```

### Push and Create PR

```bash
git push origin feature/your-feature-name
```

Open a Pull Request on GitHub with:
- Clear description of changes
- Link to related issues
- Screenshots/examples if applicable
- Test results

## Development Setup

### For AgentML Specification

Documentation and examples:

```bash
cd agentml
npm install
npm run docs:dev
```

### For agentmlx Runtime

Go development environment:

```bash
cd agentmlx

# Install dependencies
go mod download

# Run tests
go test ./...

# Build runtime
go build -o agentmlx cmd/agentmlx/main.go

# Run agent
./agentmlx run examples/chatbot.aml
```

## Testing Requirements

All contributions must include tests:

### Runtime Tests (Go)

```bash
# Unit tests
go test ./pkg/scxml/...
go test ./pkg/namespaces/...

# Integration tests
go test ./tests/integration/...

# SCXML conformance tests
go test ./tests/conformance/...
```

### Agent Tests (AgentML)

```bash
# Validate agent syntax
agentmlx validate agent.aml

# Run agent tests
agentmlx run tests/test-agent.aml
```

### Documentation Tests

```bash
# Test all code examples in docs
npm run test:docs
```

## Code Style

### Go Code

Follow Go conventions:

```bash
# Format code
go fmt ./...

# Lint code
golangci-lint run

# Vet code
go vet ./...
```

**Conventions:**
- Use `gofmt` for formatting
- Follow [Effective Go](https://golang.org/doc/effective_go.html)
- Write clear, concise comments
- Keep functions focused and testable
- Use meaningful variable names

### AgentML Files

Follow SCXML/XML best practices:

```xml
<!-- Good: Clear structure, proper indentation -->
<agentml xmlns="github.com/agentflare-ai/agentml"
       datamodel="ecmascript">

  <datamodel>
    <data id="count" expr="0" />
  </datamodel>

  <state id="idle">
    <transition event="start" target="processing" />
  </state>
</agentml>
```

**Conventions:**
- Use 2-space indentation
- Close all tags properly
- Use meaningful state IDs
- Add comments for complex logic
- Follow W3C SCXML specification

## Documentation Guidelines

Update documentation for:

- **New features**: Add examples and API docs
- **Extensions**: Document all actions and attributes
- **Breaking changes**: Update migration guide
- **Configuration**: Document environment variables
- **Examples**: Provide real-world use cases

Documentation format (MDX):

```mdx

```xml

## SCXML Conformance

AgentML aims for W3C SCXML conformance. When contributing:

1. **Check specification**: Reference [W3C SCXML spec](https://www.w3.org/TR/scxml/)
2. **Run conformance tests**: `go test ./tests/conformance/...`
3. **Document deviations**: Explain any non-conformant behavior
4. **Add tests**: Include conformance tests for new features

Current conformance: **185/193 tests passing (95.9%)**

## Creating Extensions

Contribute custom namespaces:

**Structure:**

```
my-extension/
├── README.md
├── go.mod
├── extension.go
├── actions.go
├── examples/
│   └── usage.aml
└── tests/
    └── extension_test.go
```

**Implementation:**

```go
package myextension

import "github.com/agentflare-ai/agentmlx/pkg/scxml"

type MyAction struct {
    Param string `xml:"param,attr"`
}

func (a *MyAction) Execute(ctx *scxml.Context) error {
    // Implementation
    return nil
}

func Register() {
    scxml.RegisterNamespace("github.com/example/myextension", map[string]interface{}{
        "myaction": &MyAction{},
    })
}
```

See [Custom Extensions](/extensions/custom) guide.

## Issue Reporting

When reporting issues, include:

**Bug Reports:**
- agentmlx version: `agentmlx --version`
- Operating system and architecture
- Agent file (or minimal reproduction)
- Expected vs actual behavior
- Error messages and stack traces
- Steps to reproduce

**Example:**

```
**Bug**: State transition fails in parallel states

**agentmlx version**: 0.1.0
**OS**: Ubuntu 22.04 (amd64)

**Agent file**:
[Attach minimal reproduction]

**Expected**: Both parallel states complete
**Actual**: Only first state completes

**Error**:
[Paste error output]

**Steps**:
1. Run `agentmlx run agent.aml`
2. Send event: `{"name": "start"}`
3. Observe output
```

## Feature Requests

For feature requests, describe:

1. **Use case**: What problem does this solve?
2. **Motivation**: Why is this important?
3. **Proposed solution**: How should it work?
4. **Example**: Show expected usage
5. **Alternatives**: Other approaches considered
6. **Willingness**: Can you implement this?

**Example:**

```markdown
## Feature: WebSocket IOProcessor

**Use case**: Real-time bidirectional communication for agents

**Motivation**: HTTP is request-response only. WebSocket enables
long-lived connections for chat, notifications, and streaming.

**Proposed solution**:
```xml
<send target="ws://server.com/events" type="websocket">
  <content expr="message" />
</send>
```

**Alternatives**: Server-sent events, long polling

**Willingness**: I can implement this with guidance
```

## Pull Request Process

1. **Update documentation**: Add/update relevant docs
2. **Add tests**: Include unit and integration tests
3. **Run tests**: Ensure all tests pass
4. **Update CHANGELOG.md**: Document changes
5. **Request review**: Tag relevant maintainers
6. **Address feedback**: Be responsive to review comments

**PR Template:**

```markdown
## Description
Brief description of changes

## Type of Change
- [ ] Bug fix
- [ ] New feature
- [ ] Breaking change
- [ ] Documentation update

## Testing
- [ ] Unit tests added
- [ ] Integration tests added
- [ ] SCXML conformance tests pass
- [ ] Manual testing completed

## Documentation
- [ ] Documentation updated
- [ ] Examples added/updated
- [ ] CHANGELOG.md updated

## Checklist
- [ ] Code follows style guidelines
- [ ] Self-review completed
- [ ] Comments added for complex logic
- [ ] No new warnings introduced
```

## Review Process

Maintainers will review your PR:

1. **Initial review**: Within 1 week
2. **Feedback**: Constructive suggestions
3. **Iteration**: Work together to refine
4. **Approval**: When ready, PR is approved
5. **Merge**: Maintainer merges to main

Be patient and responsive to feedback.
